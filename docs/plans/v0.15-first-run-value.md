# Plan — v0.15.0: close the "first-minute is theater" gap

## Context

Post-v0.14.0 install audit (2026-07-15, `docs/plans/plan.md` § "¿Lo instalaría si fuera un dev humano?") concluded: the tooling is technically sound but a cold-install dev bounces at minute 1 because:

1. `init` outputs `phaseTemplate` placeholders ("Stabilize project baseline") that read as project-management theater, not a real roadmap.
2. Language detection whiffs on `app.js` in the repo root (says "No language-specific stack detected"), making the tool look broken in the hello-world.
3. `add-task` succeeds silently — no ID printed, no `--json`-pipeable output, so the follow-up `update --task <id> --evidence` requires the dev to `grep` the file for the slug.
4. The killer value (agent claims done → validator disagrees) is invisible on install. The demo GIF shows two Claude sessions running, but never the *moment* where the validator catches a false `[x]`.

Each rec below is a **separate, ship-independently unit**. Order recommends 1 → 2 → 3 → 4 by impact/cost. Recs 1-3 are v0.15.0 minimum viable. Rec 4 (interactive init) is nice-to-have and overlaps with P2.2 of the prior plan.

**Success criterion for the release:** a dev who installs v0.15.0 cold on a real repo sees at least 2 tasks with concrete file references in the initial ROADMAP.md, and can chain `add-task | jq -r .task.id | xargs -I{} update --task {} --evidence …` in one shell line.

---

## Rec 1 — `init` scans the repo and generates repo-specific tasks

### Problem

Ran `roadmapsmith init --product-name MyApp --primary-user dev --problem-statement auth` on a fresh repo with `app.js` + `README.md`. Got:

```
- [ ] Stabilize project baseline and unblock high-risk delivery paths
- [ ] Reduce operational risk before v0.3
- [ ] Complete final hardening and release readiness for v1.0
```

These are `DEFAULT_CONFIG.phaseTemplates` from `src/config.js:49-62`. Zero relationship to the actual code. A dev reads this, thinks "boilerplate garbage", deletes the file.

### Root cause

`src/generator/index.js` correctly scans files (`walkFiles`, `detectLanguages`, `detectTestFrameworks`) but the P0/P1/P2 task text falls back to `phaseTemplates` when the repo is small enough that scan-derived candidates are sparse. The templates are safe defaults for "we found nothing to say" — but "we found nothing to say" should not emit 10 lines of pseudo-tasks.

### Change

**Step 1 — dead-honest empty-repo path.** When the repo scan produces fewer than N (=3) real task candidates, `init` emits a minimal ROADMAP.md with:

```markdown
<!-- rs:managed:start -->
# MyApp

**Problem being solved:** auth
**Primary user:** dev

## Detected surface
- No implementation files with clear task candidates yet.
- Detected stack: <best-effort or "unspecified">

## Your first tasks
Add a task with:
```
roadmapsmith update --add-task "Describe the first thing to build"
```
<!-- rs:managed:end -->
```

No fake phases, no fake milestones. When there's nothing to say, say nothing.

**Step 2 — real task inference on repos with signal.** When `walkFiles` finds ≥3 code files, generate tasks from *concrete signals*:

- Every function in `src/**/*.{js,ts,py,go,rs}` without a matching test file → `Add tests for <function> in <file>:<line>`.
- Every `TODO|FIXME|HACK` comment → `Address TODO: <comment text>` (file:line).
- Every dep in `package.json.dependencies` without a mention in README → `Document usage of <dep>`.
- Every `.env.example` field without value in `.env` → `Configure <ENV_VAR>`.

Cap at 10 generated tasks total. Prioritize test-coverage tasks (P0), then TODOs (P1), then docs/env (P2). If ≥3 concrete tasks land, drop `phaseTemplates` entirely.

**Step 3 — language detection fix.** `detectLanguages()` currently misses `app.js` in root. Debug: does it look only in `src/`? Fix the walk root to include the project root plus `src/`, `lib/`, `packages/*/`.

### Trade-off

- Larger surface for false positives (a function in a Rust example file that's not a real module). Mitigate: skip `examples/`, `fixtures/`, `test/` dirs (already ignored by `walkFiles`, verify).
- Some devs might want the phaseTemplates for greenfield "let me plan first" use. Preserve them behind a flag: `roadmapsmith init --with-phase-templates` for the 10% who want that.
- **Ceiling:** the inferencer is heuristic and will get some tasks wrong. That's acceptable — a task the dev deletes ("Add tests for `internalHelper()`") is far less damaging than a task that reads as corporate boilerplate. Wrong-but-concrete beats vague-but-safe.

### Verification

- Fixture test: `init` on `test/fixtures/node/` produces ≥3 tasks that name real files from the fixture.
- Fixture test: `init` on an empty git repo produces the minimal "Your first tasks" ROADMAP.md, zero phase headers.
- Regression: `init --with-phase-templates` on empty repo still emits the current v0.14.0 templated output byte-for-byte (backward compat).
- E2E: extend `scripts/e2e-smoke.sh` to grep for `<function-name>` in the generated ROADMAP.md.

## Rec 2 — Hero demo: catch the false `[x]`, don't just show two runs

### Problem

`assets/demo.gif` (referenced from root `README.md`) is a scripted A/B of two Claude sessions — one with ROADMAP.md, one without. It shows *usage*, not *value*. A visitor sees two terminals and thinks "so what". The killer moment — agent marks `[x]`, `roadmapsmith update --audit` reverts it, `checkedWithoutEvidence: 1` — is nowhere.

### Change

**Step 1 — write a deterministic reproducer** at `scripts/demo/false-claim-repro.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail
WORK=$(mktemp -d)
cd "$WORK" && git init -q && git config user.email d@d && git config user.name d
echo "// stub" > app.js && git add . && git commit -q -m init
roadmapsmith init --product-name Demo --primary-user dev --problem-statement demo --project-root .
# Simulate agent marking [x] on a task without writing code
roadmapsmith update --add-task "Add SHA-256 password hashing" --project-root . --json > /dev/null
TASK_ID=$(grep -oE 'rs:task=[a-z0-9-]+' ROADMAP.md | head -1 | cut -d= -f2)
sed -i "s|- \[ \] Add SHA-256|- [x] Add SHA-256|" ROADMAP.md
echo ">>> Agent claim on disk:"
grep "SHA-256" ROADMAP.md
echo ">>> Running audit:"
roadmapsmith update --audit --project-root . 2>&1 | head -20
echo ">>> Task after audit:"
grep -A1 "SHA-256" ROADMAP.md
```

Expected output: audit reports `checkedWithoutEvidence: 1`, and the checkbox stays `[x]` in the file but with an `⚠️ attempted but validation failed: no implementation evidence found` line appended.

**Step 2 — record it.** 15-second screencap with 3 frames:
1. Agent claims done (checkbox flips to `[x]`, no code).
2. `roadmapsmith update --audit` output showing `checkedWithoutEvidence: 1`.
3. The `⚠️ attempted but validation failed` line in ROADMAP.md.

**Step 3 — replace `assets/demo.gif`** with the new artifact. Keep the current A/B script (`scripts/demo/run.sh`) but move its GIF to `assets/demo-ab.gif` and reference it in a secondary "See it in action" subsection.

### Trade-off

- The new demo shows a failure mode, not a success — some marketing instinct will resist. That instinct is wrong for this tool: the value IS the failure catch. Lean in.
- Recording is ~2h of real work. Not something the automation can do. This is the one manual step in the plan.
- **Ceiling:** if the audit output ever changes format, the GIF goes stale. Mitigation: add a screenshot test that runs `false-claim-repro.sh` and asserts the exact `checkedWithoutEvidence: 1` line in stdout. If that test breaks, the GIF needs re-recording.

### Verification

- `bash scripts/demo/false-claim-repro.sh` exits 0 and produces the expected 3-line output deterministically across runs.
- Extend `scripts/e2e-smoke.sh` to run the repro script and grep for `checkedWithoutEvidence` in its output.
- Visual: new demo.gif embedded in README renders under 2 MB (GitHub image cap tolerance).

## Rec 3 — `add-task --json` prints the assigned ID

### Problem

```
$ roadmapsmith update --add-task "Ship login flow" --project-root . --json
{"action":"add-task"}
```

To then do `update --task <id> --evidence ...`, the dev must open `ROADMAP.md`, find the `<!-- rs:task=slug -->` marker for that specific line, copy the slug. In a pipe/script this is impossible without a hack like:

```bash
grep -oE 'rs:task=[a-z0-9-]+' ROADMAP.md | tail -1 | cut -d= -f2
```

### Change

Modify the `--add-task` code path in `bin/cli.js` to include the assigned ID + task text in the JSON output:

```json
{
  "action": "add-task",
  "task": {
    "id": "ship-login-flow",
    "text": "Ship login flow",
    "priority": "P1"
  }
}
```

Non-JSON output stays the same (`Added task: Ship login flow`) — the ID is not useful in human-facing text.

Ship as a **minor bump** — the JSON schema is being extended, not broken. Existing consumers that grep for `"action":"add-task"` still work.

### Trade-off

- ~5 lines of code change. Trivial.
- **Ceiling:** the ID slugification is deterministic (from `src/utils.js`) so a caller that regenerates the same task will get the same ID — but the JSON contract now documents that guarantee. Add a note in `roadmap-skill/README.md`.

### Verification

- Unit test: `update --add-task "X" --json` output parses as JSON with `.task.id` matching the slug of "X".
- E2E: extend `scripts/e2e-smoke.sh` to chain `TASK_ID=$(... | jq -r .task.id)` and use it in a follow-up `--evidence` call.

## Rec 4 — `roadmapsmith init --interactive` (was P2.2, promoted)

### Problem

Docs are passive. A dev reads `init --help`, invokes without flags, gets prompted for nothing, gets a file they didn't shape. First-time engagement is zero.

### Change

Add `--interactive` flag to `init`. Prompts (via `readline` stdlib, no dep):

1. Product name (default: `path.basename(process.cwd())`)
2. Primary user (default: `solo dev`)
3. Do you use AI coding agents daily? [y/n] — if n, exit with: "RoadmapSmith is designed for AI-agent workflows. `TODO.md` covers your case better."
4. Which host(s)? [claude/codex/both/none]

Non-interactive `init` behavior unchanged.

**Skipped from the original P2.2 spec:** the `--project-hours-per-week` question and the automatic hook setup. Both are premature — get the basic flow shipped first, add flags only when a user asks.

### Trade-off

- ~80 lines total (prompts + branching + defaults). Uses `readline`. No new dep.
- **Ceiling:** 4 prompts max. If wizard needs to grow beyond that, redesign — but a 10-prompt wizard is worse UX than a config file.
- Depends on Rec 1 landing first — otherwise the wizard outputs the same placeholder ROADMAP.md and adds ceremony to a broken first-run.

### Verification

- Unit test: `--interactive` with piped stdin `MyApp\ndev\ny\nclaude\n` produces the same file as `init --product-name MyApp --primary-user dev --hosts claude`.
- Unit test: answering "n" to the AI-agents question exits with code 0 (not an error, just a bail) and prints the redirect message.
- E2E: the interactive flow is NOT wired into `scripts/e2e-smoke.sh` (needs a PTY, cross-platform pain). Test locally only.

## Release strategy

- **Rec 1, 3** ship together in **v0.15.0** as a minor bump. Rec 1 is the visible value; Rec 3 is the pipeability fix.
- **Rec 2** (demo) ships alongside v0.15.0 as an assets update. Not tied to a version bump — but the README should not point to the new demo until v0.15.0 is on npm.
- **Rec 4** (interactive) ships in **v0.15.1** or **v0.16.0**. Not blocker.

**Version rationale:** v0.15.0 is a minor bump because Rec 1 changes the default `init` output shape. Anyone who scripted around the exact P0/P1/P2 phaseTemplate text needs to opt in via `--with-phase-templates`. Add a `[Breaking]` section to the CHANGELOG.

---

## What is NOT in this plan

- **P0-level rewrites of the validator.** It works. Chasing marginal false-positive reductions competes with Rec 1 for time.
- **Any distribution/marketing work.** That's Phase 2 of the prior plan (`docs/plans/plan.md` § P2). Do not start until Rec 1-3 ship — the current install experience will burn every user that a marketing push brings in.
- **`add-task` UX beyond `--json`.** No `--interactive` for add-task, no fuzzy-matching for task IDs, no templates. YAGNI.
- **A `roadmapsmith init --template=<react|node|python>` catalog.** Sounds nice, becomes 5 templates to maintain forever. If a user asks for it, revisit. Otherwise: the scan-driven inferencer (Rec 1) covers 80% of what templates would.
